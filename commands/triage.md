Every AI-generated comment posted to the issue tracker must start with:

> *Generated by AI during triage.*

## State machine

| Category | Meaning |
|---|---|
| `bug` | something is broken |
| `enhancement` | new feature or improvement |

| State | Meaning |
|---|---|
| `needs-triage` | awaiting evaluation |
| `needs-info` | waiting on reporter |
| `ready-for-agent` | fully specified, AFK-runnable |
| `ready-for-human` | needs human judgment or access |
| `wontfix` | will not be actioned |

Every triaged issue: exactly one category + one state. Conflicting states → flag and ask before acting.

## Invocation

Interpret natural language:
- "show me what needs attention" → show queue
- "look at #42" → triage that issue
- "move #42 to ready-for-agent" → quick override

## Show queue

Query and present three buckets, oldest first:
1. **Unlabeled** — never triaged
2. **needs-triage** — evaluation in progress
3. **needs-info with reporter activity since last triage** — needs re-evaluation

Show counts and one-line summary per issue.

## Triage a specific issue

**1. Gather context.** Read full issue (body, comments, labels, reporter, dates). Parse prior triage notes — don't re-ask resolved questions. Explore codebase using domain vocabulary. Check `.out-of-scope/` for prior rejections resembling this issue.

**2. Recommend.** State category + state + reasoning + relevant codebase context. Wait for direction.

**3. Reproduce (bugs only).** Trace code path, run tests or commands. Report: confirmed repro with code path, failed repro, or insufficient detail (`needs-info` signal). Confirmed repro makes stronger agent brief.

**4. Challenge (if needed).** If issue needs fleshing out, run `/pakka:challenge` on open questions.

**5. Apply outcome:**
- `ready-for-agent` → write agent brief comment (see Agent Brief Format section below)
- `ready-for-human` → same structure, note why it can't be delegated (judgment calls, external access, design decisions)
- `needs-info` → post triage notes using template below
- `wontfix` (bug) → polite explanation, close
- `wontfix` (enhancement) → write to `.out-of-scope/`, link from comment, close

## Quick state override

If maintainer says "move #42 to ready-for-agent": apply directly. Confirm what you're about to do (role changes, comment, close), then act. Skip grilling. Ask if they want an agent brief.

## Needs-info comment template

```markdown
## Triage Notes

**Established:**
- point 1

**Still needed from @reporter:**
- specific question 1
- specific question 2
```

Capture everything resolved under "Established" — work isn't lost if issue sits for weeks. Questions must be specific and actionable, not "please provide more info."

## .out-of-scope/ directory

Stores rejected enhancement concepts (not issues — one file per concept, named by concept). Check on every triage. Match by concept similarity, not keyword. "night theme" matches `dark-mode.md`.

`wontfix` enhancement → write or update file, link from comment, close issue.

## Agent Brief Format

An agent brief is posted as a comment when an issue moves to `ready-for-agent`. It is the authoritative spec an AFK agent works from. Issue body and discussion are context — the brief is the contract.

**Durable, not precise.** Describe interfaces, types, behavioral contracts — not file paths or line numbers.
**Behavioral, not procedural.** What the system should do, not how to implement it.
**Complete acceptance criteria.** Every criterion independently verifiable without human confirmation.
**Explicit scope.** State what is out of scope. Prevents gold-plating.

```markdown
## Agent Brief

**Category:** bug / enhancement
**Summary:** one-line description of what needs to happen

**Current behavior:**
What happens now. For bugs: the broken behavior. For enhancements: the status quo.

**Desired behavior:**
What should happen after the work is done. Be specific about edge cases and error conditions.

**Key interfaces:**
- `TypeOrFunctionName` — what needs to change and why

**Acceptance criteria:**
- [ ] Specific, verifiable criterion 1
- [ ] Specific, verifiable criterion 2

**Out of scope:**
What this brief does NOT cover.
```

## Red Flags

- AI comments without the disclaimer → wrong. No exceptions.
- Re-asking questions already resolved in prior triage notes → always read full history first.
- Reproducing without tracing the code path → incomplete. Code trace is part of repro.
- Agent brief without concrete, independently-verifiable acceptance criteria → incomplete. Sharpen before posting.
- Moving to `ready-for-agent` for an issue requiring human judgment → wrong state.
- Closing as wontfix (enhancement) without writing to `.out-of-scope/` → institutional memory lost.
